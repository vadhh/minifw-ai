## What Was Built

A standalone Python demo package for the finance sector, parallel in structure to `dist/minifw-usb-hospital-standalone-v2.2.0/` but with the following differences:

- HTTPS on port 8443 (hospital uses HTTP 8000)
- Local CA generated by `setup_tls.sh`, installed to system trust store and NSS databases
- `teardown_demo.sh` removes the CA after the meeting
- Scheduler writes directly to `logs/events.jsonl` using `EventWriter` — no DNS pipeline involvement
- Engine runs with `MINIFW_DNS_SOURCE=none` so it idles without touching network inputs
- Attack sequence targets `10.50.0.1` in the `trading` segment (block_threshold=80)
- All four attack steps run with 5-second delays, placing the BLOCK at approximately T+75–80s

## Design Decisions

### Why the Scheduler Bypasses the Engine

The design spec originally described the scheduler writing dnsmasq-format lines to a log file, with the engine tail-following that file through its full scoring pipeline. This was rejected in brainstorming because it creates too many failure modes: feed matching must hit the right domains, MLP inference must behave correctly, YARA rules must compile and match, and score weights must accumulate past threshold. Any one of those can fail silently, resulting in no BLOCK event during the presentation.

The chosen approach: the scheduler calls `EventWriter` directly, writing pre-scored `Event` records with exact `action`, `score`, `reasons`, and `severity` values. The dashboard reads `logs/events.jsonl` and displays whatever is there. The engine runs with `MINIFW_DNS_SOURCE=none` to idle without processing any DNS. This removes all non-determinism from the demo. The BLOCK event fires at a fixed time because the scheduler sleeps for fixed intervals.

### Why `delay=0.0` as Default Parameter

The original plan had `time.sleep(5)` inside `write_attack_sequence()` with no parameter. Running tests would hang for 20 seconds (4 steps × 5 seconds). The implementer added `delay: float = 0.0` as a parameter — tests call with no argument, `run()` calls with `delay=5.0`. This is the correct approach for any function that loops with sleeps and needs to be tested.

### Why `_CLEANED=0` Guard in `run_demo.sh`

`trap cleanup EXIT INT TERM` fires the `cleanup` function on both INT (Ctrl+C signal) and EXIT (the shell exiting after INT handling). Without a guard, Ctrl+C triggers cleanup twice, printing `[minifw] Demo stopped.` twice and violating the 4-line terminal output requirement. The guard sets `_CLEANED=1` on first entry and returns immediately on subsequent calls.

### Why Sudo Is Non-Fatal in `setup_tls.sh`

`setup_tls.sh` runs under `set -euo pipefail`. A bare `sudo cp ... && sudo update-ca-certificates` exits the script with code 1 in environments where sudo is not available or password prompt is not interactive (CI, non-root USB boot). The trust store install is wrapped in an if/else that prints `WARN` and continues. The CA install is best-effort — the demo can still run with a browser cert warning if the presenter skips that step.

## TLS Setup Details

`setup_tls.sh` generates a two-level PKI using openssl only (no mkcert, no internet):

1. CA key: `openssl genrsa 4096` → `certs/minifw-ca.key`
2. CA cert: `openssl req -x509 -days 825 -addext "basicConstraints=critical,CA:true"` → `certs/minifw-ca.crt`
3. Server key: `openssl genrsa 2048` → `certs/server.key`
4. Server CSR: `openssl req -new` with CN=localhost
5. SAN extension file: `subjectAltName=DNS:localhost,IP:127.0.0.1`
6. Server cert: `openssl x509 -req -days 825 -CA ... -extensions SAN` → `certs/server.crt`
7. System trust store: `sudo cp minifw-demo-ca.crt /usr/local/share/ca-certificates/ && sudo update-ca-certificates` — non-fatal on failure
8. NSS databases: `certutil -A -n "MiniFW Demo CA" -t "CT,,"` to `~/.pki/nssdb` and Firefox `cert9.db` files — non-fatal if certutil absent

`teardown_demo.sh` reverses steps 7 and 8, then deletes `certs/` entirely.

## Bugs Found and Fixed During Implementation

### 1. `hospital_rules.yar` in financial package (commit `16683098`)

Task 1 copied all files from the hospital standalone including `yara_rules/hospital_rules.yar`. That file contains IoMT and HIPAA-specific rules: `HIPAAPHIKeywords`, `IoMTDeviceFingerprint`, `HospitalRansomwarePattern`. These are wrong for a finance demo and would show up in the YARA scanner output during the presentation. Removed in the fix commit. `sme_rules.yar` and `test_rules.yar` were retained as they are sector-neutral.

### 2. Double-fire of `cleanup()` on Ctrl+C (commit `2cc1873f`)

`trap cleanup EXIT INT TERM` triggers on SIGINT and then again on EXIT when the shell exits. Two calls to `cleanup()` means "Demo stopped." prints twice and both kill commands run twice. Fixed with a guard variable:

```bash
_CLEANED=0
cleanup() {
    [[ "$_CLEANED" -eq 1 ]] && return
    _CLEANED=1
    kill "$ENGINE_PID"    2>/dev/null || true
    kill "$WEB_PID"       2>/dev/null || true
    kill "$SCHEDULER_PID" 2>/dev/null || true
    log "Demo stopped."
}
```

### 3. `write_attack_sequence` sleep blocking tests

Original design had `time.sleep(5)` hardcoded inside the attack sequence loop. Running the test suite would hang for ~20 seconds. Fixed by adding `delay: float = 0.0` parameter — tests call with default (no delay), `run()` calls with `delay=5.0`.

### 4. Env var pollution in test file

Module-level `os.environ.setdefault("MINIFW_SECTOR", "finance")` in the test file would persist across test files within the same pytest session, causing `test_sector_lock.py` (which uses `monkeypatch` to control `MINIFW_SECTOR`) to see stale values. Moved to an `autouse` fixture that uses `monkeypatch.setenv` — scoped per-test, reverted automatically after each test.

### 5. Health endpoint returns 307, not 200

`/health` redirects to `/health/` (trailing slash). HEALTHCHECK.sh originally checked only for HTTP 200. Updated to accept 200 or 307:

```bash
if [[ "$HTTP_CODE" == "200" || "$HTTP_CODE" == "307" ]]; then
    pass "Dashboard responds HTTP $HTTP_CODE"
```
