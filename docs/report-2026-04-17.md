## 2. Bug Fix — `api_block_rate` IndexError

**Endpoint:** `GET /admin/api/block-rate`
**File:** `app/web/routers/admin.py`

**Root cause:** Event timestamps already containing timezone info (e.g. `+00:00`) were having `+00:00` appended again, producing a future datetime. A future event results in `age < 0`, which makes `bucket_idx = num_buckets - 1 - int(age // bucket_size)` exceed 11 — causing `IndexError: list index out of range` on `buckets_total[bucket_idx]`.

**Fixes applied:**

1. Skip events with negative age (`if age < 0: continue`) — direct fix for the IndexError.
2. Harden timestamp parsing — only append `+00:00` if no timezone offset is already present; handle `Z` suffix correctly.

```python
# Before
dt = datetime.fromisoformat(ts_str.replace(" ", "T") + "+00:00")

# After
normalized = ts_str.replace(" ", "T").replace("Z", "+00:00")
if not normalized.endswith("+00:00") and "+" not in normalized[10:]:
    normalized += "+00:00"
dt = datetime.fromisoformat(normalized)
if age < 0:
    continue
```

## 4. Bug Fix — `demo.sh` crash on remote machines

**File:** `docker/demo.sh`
**Commit:** `ff81d25`

**Root cause:** `RITAPI_DIR` was evaluated unconditionally at script startup using `realpath`, which requires the path to exist. On machines without the `ritapi-adv sc/ritapi-advanced` sibling directory (any machine other than the primary dev laptop), `realpath` exits non-zero and `set -euo pipefail` kills the script — even for `./demo.sh hospital` which has nothing to do with RitAPI.

**Fix:** Moved `RITAPI_DIR` resolution inside the `api)` case block, with `2>/dev/null || true` so a missing path produces an empty string rather than an exit. The existing error message already handles the missing-directory case correctly.

```bash
# Before (line 36, runs always)
RITAPI_DIR="$(realpath "$(dirname "$0")/../../ritapi/ritapi-adv sc/ritapi-advanced")"

# After (inside api) case only)
RITAPI_DIR="$(realpath "$(dirname "$0")/../../ritapi/ritapi-adv sc/ritapi-advanced" 2>/dev/null || true)"
```

## 5. Fix — Synthesis Card Theme Compatibility

**File:** `app/web/templates/admin/dashboard.html`
**Commit:** `ce67747`

The hero area used hardcoded dark colors (`#7f1d1d → #1e293b` gradient, `#0f172a88` cells) that looked correct in dark mode but created a jarring dark island in the light-mode dashboard.

**Solution:** Extracted hero styling into CSS classes with `[data-bs-theme="dark"]` overrides.

| Element | Light mode | Dark mode |
|---------|-----------|-----------|
| Hero background | `linear-gradient(#fee2e2, #fff1f2)` soft red | `linear-gradient(#7f1d1d, #1e293b)` dramatic dark |
| Hero title color | `#dc2626` | `#f87171` |
| Hero source color | `#9f1239` | `#fca5a5` |
| 2×2 cells | `var(--mf-surface-2)` + `var(--mf-border)` | `#0f172a88`, no border |
| Kernel/AI colors | Theme-inherited | Theme-inherited |
