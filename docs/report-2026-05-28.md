## 1. MINIFW-AI Healthcare

Healthcare was first. The product already had a working standalone Python demo
(`bash run_demo.sh`, port 8000, HTTP, no Docker). What was missing was a GUIDE.md
and a root-level `build_deb.sh`.

### GUIDE.md

Created `dist/minifw-usb-hospital-standalone-v2.2.0/GUIDE.md` (295 lines). Nine
sections: Overview, Prerequisites, Quick Start, Dashboard Walkthrough, Threat
Scenarios, Configuration, Admin Reference, Troubleshooting, Production Deployment.

The guide was written against the actual source files — `run_demo.sh`, `policy.json`,
and the demo data files — not from memory. This caught several inaccuracies that went
through multiple review cycles before the final version was correct:

- The terminal output block had `[minifw-demo]` as the log prefix. The real prefix
  from `run_demo.sh` is just `[minifw]`. Fixed.
- The config path in the guide said `config/modes/minifw_hospital/policy.json`.
  The standalone package has a flat layout — it's just `config/policy.json`. Fixed.
- The decision threshold section showed a flat ALLOW/MONITOR/BLOCK table (0–39,
  40–69, 70–100). The real `policy.json` has per-segment thresholds — `default`
  block=85, `mednet` block=45, `iomt` block=60, etc. The flat table was wrong.
  Replaced with the actual segment structure from the file.
- Log file referenced as `logs/minifw_engine.log`. Actual file is `logs/engine.log`.
- The policy.json excerpt showed `thresholds` and `score_weights` as top-level keys.
  Those keys don't exist. The real structure uses `segments` and `features`. Fixed.

Three commit rounds to get it clean. Could have been one if the first draft had been
read from the file rather than from assumptions about the structure.

`README.txt` and `INSTALL.md` were both replaced with short stubs pointing to
GUIDE.md. They had duplicated content that's now consolidated.

### build_deb.sh

`CLAUDE.md` documents `bash build_deb.sh hospital` as the way to build a .deb, but
the script only existed at `scripts/build_deb.sh`, not the repo root. The wrapper:

```bash
#!/usr/bin/env bash
exec "$(dirname "$0")/scripts/build_deb.sh" "$@"
```

Three lines. `scripts/build_deb.sh` already handles all six sectors correctly
(`hospital education government finance legal establishment`), so this was just a
matter of making the documented interface work. Validated by passing an invalid sector
(confirms the error message fires) and `hospital` (confirms the build header prints
with the right sector).

## 2. MINIFW-AI Financial

### The problem: LIVE was broken

The financial standalone package at `dist/minifw-usb-financial-standalone-v2.2.0/`
was essentially empty — just a few leftover files. Every runtime file was gone.

The culprit was commit `c2eeba12` with the message "Cleanup: Menghapus file yang
seharusnya diabaikan oleh .gitignore" (cleaning up files that should be ignored by
.gitignore). Since `dist/` is in `.gitignore`, whoever ran that cleanup wiped the
entire financial standalone package. The files had been force-added (`git add -f`)
when they were committed, but the cleanup pass didn't know that and removed them.

Recovery was straightforward once the cause was identified. All 33 files were still
in the git object store, accessible via `git show c2eeba12^:<path>` (the parent
commit of the deletion). Files restored in one commit:

- Core scripts: `run_demo.sh`, `fast_reset.sh`, `setup_tls.sh`, `teardown_demo.sh`,
  `HEALTHCHECK.sh`, `recover_demo.sh`
- Config: `config/policy.json`, all 14 files under `config/feeds/` (deny domains,
  tor exit nodes, financial fraud list, crypto scam list, etc.)
- Demo data: `demo_data/normal_traffic.json`, `demo_data/attack_traffic.json`
- Model: `models/mlp_model.pkl` (96KB pre-trained MLP)
- YARA: `yara_rules/sme_rules.yar`, `yara_rules/test_rules.yar`
- Python package marker: `scheduler/__init__.py` (0 bytes — correct for a package)
- Docs: `INSTALL.md`, `README.md`, `PRESENTER_CARD.md`, `DEMO_SCRIPT.md`, `RECOVERY.md`

### GUIDE.md

Created `dist/minifw-usb-financial-standalone-v2.2.0/GUIDE.md` (443 lines — the
longest of all the guides because the financial demo has the most complex setup).

Key things the financial guide covers that the others don't:

- `setup_tls.sh` must be run before the first `run_demo.sh`. The demo runs on HTTPS
  (port 8443) with a self-signed cert. If you skip TLS setup, the server won't start.
  The guide makes this the first step in Quick Start.
- The demo scenario is ArborCrest Capital, a fictional investment firm. The guide
  explains the network layout: trading floor on `10.50.0.0/24`, ERP/back-office on
  `192.168.1.0/24`, each with different block thresholds.
- The attack chain is a 4-step progression that plays out over multiple injector
  loops: step 1 is a Tor exit node query from a trading workstation (score=55,
  MONITOR), step 2 is a C2 beacon that escalates to score=70 (MONITOR edge), step 3
  escalates to score=78 (BLOCK on trading, threshold=75), step 4 is the same domain
  from a different source but score=95 (hard BLOCK, deny list hit).

`README.txt` replaced with a stub. The old README.md was kept as-is since it had
presenter notes that aren't in GUIDE.md.

## 3. MINIFW-AI Schools

Schools is Docker-based (unlike hospital and financial which are Python standalone).
Port 8447, HTTPS, `admin / Education1!`. The injector container writes dnsmasq-format
log lines to a shared volume at `/logs/dnsmasq.log`, which the engine container reads.

### GUIDE.md

Created `dist/minifw-usb-education-v2.2.0/GUIDE.md` (282 lines). The segment
threshold table was read directly from `config/modes/minifw_school/policy.json`:

| Segment | MONITOR | BLOCK | Subnet |
|---------|---------|-------|--------|
| student | 35 | 70 | 10.10.0.0/16 |
| guest | 30 | 60 | 192.168.100.0/24 |
| staff | 50 | 80 | 192.168.1.0/24 |
| admin | 55 | 85 | 10.0.0.0/24 |

The injector runs 8 emit() calls per loop plus a burst sequence. Domains injected:
`khanacademy.org`, `bbc.co.uk`, `wikipedia.org` (all ALLOW), `instagram.com` (MONITOR,
student), `nordvpn.com` (MONITOR), `nordvpn-bypass.proxy.io` (BLOCK, score=75, YARA
VPN proxy rule), `filter-bypass.student.io` (BLOCK, guest, score=75), `bet365.com`
(MONITOR, student). Followed by 200 burst queries to `nordvpn-bypass.proxy.io`.

Two accuracy issues caught during review: the initial draft said "6 scenario events"
in the Quick Start (copied from the plan) but counting the `emit()` calls in inject.py
gives 8. Also the loop timing was written as "~33 seconds" — adding up the actual
`time.sleep()` calls in inject.py (1+1+2+2+2+2+2=12s between emits, 5s after burst,
10s inter-loop) gives ~27 seconds. Both corrected.

`README.txt` replaced with a stub.

## 4. MINIFW-AI Legal

Legal is also Docker-based. Port 8448, HTTPS, `admin / Legal1!`. The scenario is a
law firm (Clifton & Associates) with 5 network segments.

### GUIDE.md

Created `dist/minifw-usb-legal-v2.2.0/GUIDE.md` (285 lines). Segment thresholds from
`config/modes/minifw_legal/policy.json`:

| Segment | MONITOR | BLOCK | Subnet | Rationale |
|---------|---------|-------|--------|-----------|
| partner | 55 | 85 | 10.20.0.0/24 | Senior counsel, most trusted |
| associate | 45 | 72 | 10.20.1.0/24 | Junior lawyers, standard |
| paralegal | 38 | 70 | 10.20.2.0/24 | Support staff, stricter |
| client | 30 | 62 | 192.168.200.0/24 | Meeting rooms, unknown devices |
| guest | 28 | 60 | 192.168.100.0/24 | Visitor WiFi, tightest |

The associate block threshold is 72 (not the default 70 or 80) because the demo
domain `clio-encrypt.c2-server.ru` scores exactly 75 (feed+40, YARA+35), and the
threshold needs to sit below 75 for the block to fire cleanly. The `_note` in
policy.json explains this explicitly.

Injector runs 7 emit() calls per loop: 3 ALLOW (westlaw, lexisnexis, courts.gov),
1 MONITOR (wetransfer-legal.io, paralegal, score=40), 3 BLOCK (Tor exit relay from
client room, ransomware C2 from associate net, privilege breach from paralegal), then
the 200-request burst on clio-encrypt. Loop cycle ~27 seconds.

The README.txt didn't previously exist in this package — created fresh as a stub.

## 5. RITAPI V-Sentinel

V-Sentinel is a two-service stack: Django ops dashboard (port 8000) + MiniFW-AI web
engine (port 8080), backed by PostgreSQL and Redis. The demo is fully Docker-based.
`demo.sh` starts the stack, waits for Django to be healthy, seeds demo data, and
prints the access info. No attack script runs automatically — you just browse the
pre-seeded dashboard.

### GUIDE.md

Created `dist/usb-vsentinel/GUIDE.md` (293 lines). Login is `admin / admin123`.
Demo data is seeded by `demos/demo_ctl.sh seed` which runs Django management commands
inside the container — 20 alerts across 10 threat types (brute_force, sql_injection,
xss_attempt, port_scan, dos_attempt, credential_stuffing, path_traversal,
suspicious_user_agent, rate_limit_exceeded, tor_exit_node), 100 request log entries
with ML classification labels, 30 MiniFW DNS events, plus ASN and IP reputation data.

The reset workflow is `demos/demo_ctl.sh reset` then `demos/demo_ctl.sh seed`. This
is different from the minifw packages (which use `fast_reset.sh`) so the guide makes
it explicit. The old README.txt covered this but in the quick-reference style. The
new GUIDE.md expands it into context — what each section of the dashboard shows, what
the alert severity tiers mean, what the ML labels indicate.

### build_deb.sh

V-Sentinel had no `build_deb.sh` and no packaging directory to work from. Wrote one
from scratch. It bundles `projects/` (Django + MiniFW source), `scripts/`, and
`install.sh` into `/opt/ritapi-vsentinel/` inside the staging tree. The postinst
just calls `bash /opt/ritapi-vsentinel/install.sh install`. The package ends up
at ~102MB.

## 6. RITAPI Advanced

RITAPI Advanced is a FastAPI layer-7 API protection service — WAF, rate limiting,
bot detection, tenant isolation, injection/exfiltration detection. Port 8001 (HTTP).
The demo runs an automated 5-act attack suite after startup.

### GUIDE.md

Created `dist/ritapi-usb-v1.4.0/GUIDE.md`. The five attack acts from `demo_attack.sh`:

- **Act 1** — Baseline: valid JWT (tenant=acme) + matching X-Target-ID header → 200
- **Act 2** — Tenant isolation: two mismatch variants (acme token claiming corp, and
  corp token claiming acme) → both 403
- **Act 3** — Injection WAF: 5 payloads from 5 distinct `X-Forwarded-For` IPs —
  UNION SELECT SQLi, `<script>alert(document.cookie)</script>` XSS, `/etc/passwd; id`
  command injection, `../../../etc/shadow` path traversal, SQLi in POST body →
  all 403
- **Act 4** — Bot detection: `sqlmap/1.7.8#stable` and `Nikto/2.1.6` user-agent
  strings, 4 requests each from distinct IPs, risk score accumulates → 403 by
  request 4
- **Act 5** — Rate limiting: 55 rapid GET /dashboard requests from `10.0.flood.1`,
  limit is 50/60s → 429 at request 51

Credentials for the admin token API: `X-Admin-Secret: demo-admin-ritapi-2024`.
Dashboard at `/dashboard`, metrics at `/metrics` (Prometheus format).

### build_deb.sh

RITAPI Advanced already had a `packaging/` directory with a proper DEBIAN structure
(control file with VERSION_PLACEHOLDER and SIZE_PLACEHOLDER, postinst that creates the
`ritapi` system user and enables `ritapi-advanced.service`, prerm, postrm, systemd
units, doc directory). So `build_deb.sh` just needed to:

1. Copy the `packaging/` template into the build directory
2. Patch the placeholders with `sed`
3. Copy the app source tree (`app/`, `configs/`, `rules/`, `scripts/`, `requirements.txt`)
4. Copy the config skeleton to `/etc/ritapi-advanced/` (routing.yml + 3 policy files)
5. Run `dpkg-deb --build`

First run failed because the `build/` parent directory didn't exist before the first
`cp -r packaging "${BUILD_DIR}"`. Added `mkdir -p "$(dirname "${BUILD_DIR}")"` before
the copy. Second failure was `chmod 600` on the policies directory — made the directory
unreadable, which caused `rm -rf` to fail on the next run. Changed to `chmod 755` on
the directory and `chmod 644` on the files inside. Third run produced a clean 220KB
`.deb`.

The `.deb` is source-only — the Python venv is built at install time by postinst, not
bundled. This keeps the package small but means the target host needs internet access
and python3-pip to install.
