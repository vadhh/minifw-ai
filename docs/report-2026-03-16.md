## 2. Critical Issues — Resolved

### C-1: VSentinel GAMBLING_ONLY Hard Guard
**Finding:** `app/minifw_ai/main.py` contained a guard that crashed the daemon unless
`GAMBLING_ONLY=1` was set — bespoke code from a gambling deployment preventing any
other sector from running.

**Resolution:**
- Guard removed from `main.py` (lines 4–6)
- `Environment=GAMBLING_ONLY=1` removed from both `minifw-ai.service` and `minifw-ai-web.service`
- `scripts/vsentinel_scope_gate.sh` rewritten to validate against the canonical sector
  list (`hospital education government finance legal establishment`) and reject `gambling`
- Embedded in `DEBIAN/postinst` to prevent recurrence on reinstall

---

### C-2: Prometheus Metrics Exposed on 0.0.0.0:9090
**Finding:** `start_metrics_server()` called `start_http_server(port)` with no address
binding — metrics endpoint reachable from any interface.

**Resolution:**
- `prometheus/metrics.py`: `start_metrics_server(port, addr="127.0.0.1")` — default
  binding changed to localhost
- Verified: `ss -tulnp` confirms `127.0.0.1:9090`

---

### C-3: Grafana Accessible on 0.0.0.0:3000
**Finding:** Default Grafana install listens on all interfaces.

**Resolution:**
- `http_addr = 127.0.0.1` injected into `/etc/grafana/grafana.ini` `[server]` section
- Grafana restarted; verified: `127.0.0.1:3000`
- Step embedded in `DEBIAN/postinst` step 7

---

### C-4: ipapi_guard TCP Listener on 0.0.0.0:5514
**Finding:** `/opt/ipapi_guard/ipapi_guard.py` hardcodes `server.bind(("0.0.0.0", port))`;
no `--bind` CLI argument exists.

**Resolution:**
- Source patched: `server.bind(("0.0.0.0", port))` → `server.bind(("127.0.0.1", port))`
- Service restarted; verified: `127.0.0.1:5514`

---

### C-5: nginx Reverse Proxy on 0.0.0.0:7004
**Finding:** `/etc/nginx/sites-available/forward-7003.conf` used `listen 7004;` — bound
to all interfaces.

**Resolution:**
- Config patched: `listen 7004;` → `listen 127.0.0.1:7004;`
- `systemctl restart nginx` required (reload insufficient to drop old socket)
- Verified: `127.0.0.1:7004`

## 3. Warnings — Mitigated or Documented

### W-1: Wired NIC Not Yet Active
`enp1s0 / enp3s0 / enp4s0` detected but no cable connected. Engine is currently
processing only loopback + `enp0s3` traffic. **Action required:** plug ethernet cable

---

### ~~W-2: Flow Tracking Degraded (kernel 6.8)~~ — RESOLVED

`stream_conntrack_flows()` now auto-detects whether `/proc/net/nf_conntrack` exists
and falls back to the `conntrack -L` CLI when it is absent (kernel 6.8,
`CONFIG_NF_CONNTRACK_PROCFS=not set`). The CLI output format is identical to the procfs
format — `parse_conntrack_line()` is reused unchanged. `conntrack` is now a declared
`.deb` dependency.

Hard threat gates (burst flood, bot detection, flow frequency) are **fully operational**
on kernel 6.8. `nf_conntrack_netlink` kernel module provides the data source via the CLI.

---

### W-3: CUPS Print Service Was Enabled
CUPS and `cups-browsed` were active on a firewall appliance — unnecessary attack surface.

**Resolution:** Disabled and masked via `postinst` step 8:
```bash
systemctl stop cups cups-browsed
systemctl disable cups cups-browsed
```

---

### W-4: daemon_stop Not Written on SIGTERM
`audit_daemon_stop()` is called only in the `KeyboardInterrupt` handler. `systemctl stop`
sends `SIGTERM`, which exits the process without writing the stop audit event. All
`daemon_start` events are confirmed in the audit log; no corresponding stop events exist.

**Impact:** Low — audit trail gap only; enforcement and logging are not affected.
**Mitigation path:** Add `signal.signal(signal.SIGTERM, handler)` in `main.py`. Tracked
in `TODO.md`.

---

### W-5: YARA Rules — Test Rules Only
Production deployment uses `yara_rules/test_rules.yar` (bundled test rules for
GamblingKeywords, MalwarePatterns, ApiAbuse). These provide functional coverage but are
not a full threat intelligence ruleset.

**Recommendation:** Replace with production-grade YARA rules (e.g., YARA-Forge, Sigma
YARA, or custom rules) before go-live. Set `MINIFW_YARA_RULES` env var to point to the
production rules directory.

## 8. Out-of-Box Installation

All hardening steps are embedded in `DEBIAN/postinst`. A clean `dpkg -i minifw-ai_2.0.0_amd64.deb`
on a fresh Ubuntu 24.04 system will automatically:

1. Create Python venv and install dependencies
2. Generate `MINIFW_SECRET_KEY` and `MINIFW_ADMIN_PASSWORD` (stored in `/etc/minifw/minifw.env`)
3. Generate self-signed TLS certificate (`/etc/minifw/tls/server.crt`)
4. Create admin user in SQLite database
5. Load `nf_conntrack` kernel module + persist to `/etc/modules-load.d/`
6. Restrict Grafana to `127.0.0.1` if installed
7. Disable CUPS
8. Enable and start `minifw-ai` and `minifw-ai-web` services

No post-install scripts or manual steps required.
