## Codebase Analysis

### Initial Discovery
The repository at `/home/sydeco/minifw-ai/` contained four directories that appeared to be variants of the same product:

```
(BASE)minifw-ai/minifw-ai/       ← FastAPI engine
minifw-ai_hospital/              ← FastAPI engine (copy) + .deb packaging
minifw-ai_education/             ← Django security platform
minifw-ai_establishment/         ← Django platform (superset of education)
```

The first signal that these were not simple flavors was the size difference: the BASE/hospital directories each weighed roughly 2–3 MB of Python while education/establishment each exceeded 50 MB due to bundled installers, legacy agents, and a full Django + Flask stack.

### Variant-by-Variant Breakdown

**BASE (`(BASE)minifw-ai/minifw-ai/`)**
The cleanest and most modern codebase. A pure Python FastAPI daemon that reads dnsmasq/Zeek logs and makes block/monitor/allow decisions via a multi-stage pipeline:
- `main.py` — event loop, hardcoded hard-threat gates (PPS saturation, burst flood, bot patterns)
- `policy.py` / `feeds.py` — config and blocklist management
- `burst.py` — sliding-window DNS query rate tracking
- `collector_flow.py` — 24-feature ML vector builder from 5-tuple flows
- `utils/mlp_engine.py` — scikit-learn MLPClassifier inference
- `utils/yara_scanner.py` — YARA pattern matching (had the compile bug, see blockers)
- `sector_lock.py` — immutable singleton that bakes sector identity at deploy time
- `sector_config.py` — per-sector threshold adjustments and extra feed lists

The web dashboard (FastAPI + SQLAlchemy + SQLite) ran as a separate subsystem from the same install. No TLS MITM — all inspection happens at DNS/SNI metadata level only.

**Hospital (`minifw-ai_hospital/`)**
Byte-for-byte identical to BASE engine code. The only additions were `.deb` packaging (`build_deb.sh`, `packaging/deb/DEBIAN/`, `systemd/`). Confirmed by diff — zero engine-level differences. The hospital-specific IoMT alerting logic was actually embedded inline in BASE's `main.py` rather than isolated in a module, which was the first thing to extract.

**Education (`minifw-ai_education/minifw-ai_education/`)**
A full-stack Django application — fundamentally a different architecture. Key components:
- Django apps: `firewall/`, `ips/` (Snort 3), `hids/` (OSSEC), `agents/`, `router/`
- `services/minifw/engine.py` — the security rule engine. Contained `evaluate_general()`, `supply_chain_guard()`, and `evaluate_education()` — the three functions worth porting
- A Flask dashboard (`dashboard/`) proxied by Nginx
- Agent2 (log monitor), Agent4 (TensorFlow ML model + TCP RST injection via raw socket)
- `services/agent/yaraagent/yara/` — 292 YARA rule files organized into 11 threat categories
- A hardcoded MAC address in `engine.py` lines 15–18 — identified as a development artifact from the previous developer (used to restrict the engine to a specific test machine)

Critical discovery: the Django variant's enforcement was **silently broken**. The systemd unit used `CapabilityBoundingSet=~all` which strips all Linux capabilities from the process, including `CAP_NET_ADMIN` which is required for iptables/nftables operations. Every `iptables -A` call in the codebase was failing silently with permission denied. The dashboard showed rules as "applied" but no enforcement was actually happening.

**Establishment (`minifw-ai_establishment/`)**
Education plus a `vpn/` Django app (OpenVPN management), Cowrie SSH honeypot integration, network segmentation scripts, and additional `.deb` package scripts. The `engine.py` equivalent for establishment was a single comment line — no rules had been implemented by the previous developer. The Cowrie honeypot IP and segment-based VPN rules only existed in the installer scripts as configuration stubs, not in the enforcement engine.

### Architecture Comparison

| Dimension | BASE / Hospital | Education / Establishment |
|---|---|---|
| Runtime | FastAPI daemon | Django + Flask + Nginx |
| Enforcement | nftables via ipset (works) | iptables calls (silently broken) |
| ML model | scikit-learn MLP (24 features) | TensorFlow Agent4 (raw socket, broken) |
| YARA | 1 monolithic rules.yar (had compile bug) | 292 categorized files (correct structure) |
| Sector rules | Inline in main.py (hospital only) | engine.py evaluate_* functions |
| Packaging | .deb (hospital only) | .deb (establishment only) |
| Maintainability | High — clean module boundaries | Low — monolithic, multiple broken layers |

### Architecture Decision

Three paths were considered:

**Path A — Fix the Django variants**: Repair the `CapabilityBoundingSet`, replace Agent4's raw socket with a proper netlink approach, port the Flask dashboard properly. Estimated scope: 3–4 weeks, high risk of introducing new breakage in already complex code.

**Path B — Keep variants separate**: Continue developing BASE and Django variants independently, accept code duplication. Risk: security rule improvements in BASE never reach education/establishment deployments and vice versa.

**Path C — Consolidate on FastAPI BASE**: Treat BASE as the single source of truth. Extract security rule knowledge from Django `engine.py` into `sector_rules/<sector>.py` modules. Discard Django enforcement entirely. Build all six sector `.deb` packages from the same BASE source tree.

**Path C was chosen** because: the FastAPI engine's enforcement actually works (nftables + ipset); the valuable parts of the Django variants are the security rules (not the architecture); all six sectors can share the same tested, maintained codebase; sector behavior is controlled by swappable modules not separate forks.

## Findings

### Architecture
- The six sector modules now form a complete, consistent pipeline. Every sector that `build_deb.sh` can package now has a corresponding `sector_rules/<sector>.py` that runs in the decision pipeline.
- The `SectorType` enum key (`SCHOOL`) was intentionally kept while the value was changed to `"education"`. This avoids breaking any internal Python code that references `SectorType.SCHOOL` while ensuring the runtime sector string matches the product naming.
- `sector_lock.py` derives its valid sector list from the `SectorType` enum values at runtime (`[s.value for s in SectorType]`), so the rename propagated automatically — no hardcoded list to update.

### New sector modules
- **government.py**: operates at DNS/TLD level — no Geo-IP database required. Country-code TLD matching (`.ru`, `.cn`, `.ir`, `.kp`) is a practical heuristic that covers the majority of nation-state threat traffic without needing an IP→country lookup library. APT deep-subdomain heuristic catches DNS tunnelling tools (dnscat2, iodine) which encode C2 payloads across many subdomain labels.
- **finance.py**: Tor `.onion` detection is deterministic and zero false-positive at DNS level. Crypto phishing detection uses keyword patterns on the full domain string (not just TLD), catching typosquats like `binance-login.net` and `metamask-connect.io`.
- **legal.py**: DNS exfiltration detection targets the two main data encoding strategies used by DNS tunnel tools — long labels (> 50 chars, base64-encoded chunks) and deep chains (> 6 labels, data encoded across multiple levels). Paste-site blocking is deterministic and high-confidence for legal confidentiality compliance.

- Sector_rules pipeline: 22 inline verification cases (8 from session 1, 14 from today) all pass.

## Blockers Encountered

### 1. `yara.compile(filepaths=dict)` silently drops files (session 1)
**Problem**: The YARA Python API's `filepaths` parameter only accepts one file per namespace key. When multiple files share a namespace (e.g. all `ransomware/` rules), only the last one was loaded. No error, no warning — 200+ rules silently discarded.
**Solution**: Changed to reading file content manually, concatenating per namespace, and using `yara.compile(sources=dict)`. Now all 293 files load correctly.

### 2. VPN policy conflict between base and sector modules (session 1)
**Problem**: `base.supply_chain_guard()` originally contained a VPN keyword detection block. This fired before `education.py` could return "block" and before `establishment.py` could return "allow" for trusted segments. The pipeline stopped at the first non-allow result, so sector-specific VPN logic was effectively dead code.
**Solution (two-part)**:
- Removed VPN detection entirely from `base.supply_chain_guard()` — VPN policy is documented as sector-specific by design.
- Changed pipeline from "stop at first non-allow" to "run all three stages, keep most severe result" using `_sev = {"block": 2, "monitor": 1, "allow": 0}`.

### 3. Shannon entropy threshold miscalibration (session 1)
**Problem**: The legacy code used entropy threshold 4.0 calibrated for full URLs. At the DNS label level (leftmost label only), normal labels score 2.5–3.5 and DGA labels score 3.5–4.5. Threshold 4.0 missed many DGA domains.
**Solution**: Lowered threshold to 3.5 with minimum label length of 12 characters to suppress false positives on short legitimate labels. Added documentation explaining the calibration difference between label-level and URL-level entropy.

### 4. Bash shell errors with parentheses in directory name (session 1)
**Problem**: The directory `(BASE)minifw-ai` contains parentheses which are special characters in bash, causing `mv`, `cp`, `find` commands to fail even when quoted.
**Solution**: Used `python3 -c "from pathlib import Path; ..."` for file operations in that directory tree instead of bash builtins.

### 5. `sqlalchemy` not installed in test environment (today)
**Problem**: `test_sector_lock.py` imports `from app.models.user import SectorType` which triggers `from sqlalchemy import ...` at module load. Running pytest without the web stack dependencies installed caused all 20 tests to fail immediately.
**Solution**: Installed `sqlalchemy` via pip3. The issue only affects the web-stack-dependent tests — the engine tests (`sector_rules` inline tests) use only stdlib and do not require sqlalchemy.

## Thought Process

### On sector naming ("school" vs "education")
The sector identifier is user-facing in two places: the `MINIFW_SECTOR` environment variable and the `.deb` package name. "school" is too narrow — the education sector package is deployed in universities and vocational colleges, not only K-12. Renaming to "education" aligns the identifier with the sector's actual scope. The enum key (`SectorType.SCHOOL`) was kept to minimize internal refactoring risk — only the runtime string value matters for sector dispatch.

### On DNS-layer geo-IP for government sector
A full Geo-IP solution requires an IP-to-country mapping database (e.g. MaxMind GeoLite2), which needs periodic updates and introduces a runtime dependency. At the DNS level, country-code TLD matching covers a large portion of nation-state threat infrastructure without any external database. The approach is intentionally imperfect — it is a first-pass heuristic that flags traffic for review, not a definitive geo-fencing mechanism. A real Geo-IP lookup belongs in the HTTP inspection layer when that collector is added.

### On not duplicating base.py rules in sector modules
`base.py` already blocks sensitive payment APIs (Stripe, PayPal) over non-TLS. `finance.py` extends this with a broader set of finance-specific APIs (Visa, Mastercard, Plaid, Binance) rather than duplicating the base check. The most-severe-wins pipeline ensures both checks run — finance.py adding more domains is purely additive.

### On legal.py using "monitor" not "block" for file sharing
Legal workflows legitimately use services like WeTransfer for discovery file exchange with opposing counsel or with clients. Blocking these outright would create false positives that impede real work. Monitoring captures the event for audit review without disrupting operations — consistent with the `monitor_threshold_adjustment: -10` in the legal sector config (more sensitive monitoring, not more aggressive blocking).
