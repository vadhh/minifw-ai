## What Was Done

### 1. Financial Executive Demo — Full Build

Transformed the financial standalone package (`dist/minifw-usb-financial-standalone-v2.2.0/`) from a bare technical demo into a presentation-ready executive demo. The work covered five areas.

**Demo narrative — ArborCrest Capital**

Replaced the generic financial domain names with a consistent fictional firm: ArborCrest Capital, a $4B wealth management company. The firm has two network segments used throughout the demo: a trading floor (`10.50.0.x`) running Bloomberg, Reuters, SWIFT, Refinitiv, NASDAQ, and an internal ERP layer (`192.168.1.x`) running Oracle ERP and SAP.

This gives the presenter something concrete to point at instead of generic IP addresses.

**Attack sequence — 5-step escalation**

The attack chain in `demo_data/attack_traffic.json` and `scheduler/demo_scheduler.py` was rewritten to tell a coherent story:

| Step | T+ | Domain | Score | Action | What it represents |
|------|----|--------|-------|--------|--------------------|
| 1 | 90s | `tor-exit-4f2a.net` | 55 | monitor | Attacker probes from trading floor using anonymizer |
| 2 | 96s | `c2.trickbot-gate.com` | 72 | monitor | TrickBot C2 beacon — banking trojan phoning home |
| 3 | 102s | `exfil.payment-collect.io` | 82 | monitor | Lateral move — attacker pivots to Oracle ERP subnet |
| 4 | 108s | `exfil.payment-collect.io` | 89 | monitor | Escalating exfil — client portfolio data |
| 5 | 114s | `exfil.payment-collect.io` | 95 | **block** | PCI-DSS violation threshold crossed — hard block |

Step 3 (the ERP pivot) was the main addition over the original 4-step chain. It shows lateral movement rather than just a straight attack, which is more realistic and gives the presenter more to explain.

Trace IDs for normal events use `TXN-AC-{hex}` format. Attack events use `SWIFT-MT103-{hex}`. The format makes events look like financial transaction references on the dashboard.

**Timing**

Phase 1 (normal traffic): 0–90s, one event every ~11 seconds, scores 18–22.
Phase 2 (attack): 90–114s, one step every 6 seconds.
Phase 3 (post-block normal): resumes from 120s.

**Presenter materials**

Two documents were created:

`DEMO_SCRIPT.md` — full 7-phase script with timing markers, screen cues, word-for-word say-lines, and explanations of what each event means in business terms. Also includes four Q&A pairs covering the most likely questions (false positives, agent vs. rule-based, compliance, deployment).

`PRESENTER_CARD.md` — one-page cue card. An 8-row table (phase / T+ / score / screen cue / what to say) and a 6-row key numbers table (block threshold, time to block, industry average detection time, data exfiltrated, human interventions, PCI-DSS status).

**Fast reset — `fast_reset.sh`**

Wrote a recovery script targeting sub-60s demo restart between back-to-back meetings. It:
- Kills engine, web, and scheduler processes in parallel (not sequentially like the old `recover_demo.sh`)
- Force-clears port 8443 if still held
- Deletes `logs/events.jsonl` only — preserves `minifw.db` so the admin user stays provisioned
- Relaunches `run_demo.sh` in background
- Health-polls at 0.5s intervals with a 45s timeout and reports elapsed time

The curl command in the health poll uses `--max-time 2 --connect-timeout 1` so a hung connection doesn't defeat the timeout. This was caught in code review and fixed before merge.

Actual recovery time in testing: 8–12 seconds.

### 2. Venv Auto-Activation Fix

**Problem:** When pulling the repo on a fresh machine and running `bash run_demo.sh`, Python immediately fails with `ModuleNotFoundError: No module named 'sqlalchemy'`. The venv is not committed to git, so a fresh clone has no active Python environment.

**Root cause:** The scripts assumed the caller had already activated the venv. Nothing in the scripts did it automatically.

**Fix:** Added one line to each shell script, immediately after `cd "$SCRIPT_DIR"`:

```bash
[[ -f venv/bin/activate ]] && source venv/bin/activate
```

This silently no-ops if venv doesn't exist (so first-run setup instructions still work) and activates it automatically if it does.

Applied to:
- `dist/minifw-usb-financial-standalone-v2.2.0/run_demo.sh`
- `dist/minifw-usb-financial-standalone-v2.2.0/fast_reset.sh`
- `dist/minifw-usb-hospital-standalone-v2.2.0/run_demo.sh`
- `dist/minifw-usb-hospital-standalone-v2.2.0/recover_demo.sh`

`INSTALL.md` (financial) was also updated to correct stale timing references (T+60s → T+90s, 4-step attack → 5-step).

`README.md` (hospital) was updated to add a "First time on a fresh machine" section showing the venv setup commands before `bash run_demo.sh`.

`requirements.txt` for the hospital standalone was not tracked in git. Added it so fresh clones have everything they need.
